BLOQUE 01

El jugador como diccionario

estructuras de datos · funciones · return
✓ completado
Objetivo ¿Por qué un diccionario y no variables sueltas? ▼
👨‍🏫
Imaginen que tienen que pasarle la información del jugador a una función. Con variables sueltas tendrían que hacer esto: atacar(nombre, vida, ataque, oro, inventario). Cinco parámetros para una sola cosa. Y si mañana agregan "experiencia", tienen que modificar cada función del programa. Con un diccionario, la función recibe un solo objeto que representa al jugador completo. Agregar un campo no rompe nada más.
En programación, agrupar datos que pertenecen juntos se llama cohesión. Un jugador tiene nombre, vida, ataque, oro — todo eso junto forma una unidad. El diccionario es la forma más natural de expresar eso en Python.
Desarrollo Escribe la función crear_jugador ▼
👨‍🏫
Primero escriban esto arriba del todo en su archivo. Las constantes van primero, porque son la "configuración" del juego. Si mañana quieren cambiar la vida inicial de 20 a 30, lo cambian en un solo lugar.

# ═══════════════════════════════════════════
# CONSTANTES DEL JUEGO
# ═══════════════════════════════════════════
HP_INICIAL    = 20
ATK_INICIAL   = 5
SALAS_TOTALES = 3

# ═══════════════════════════════════════════

def crear_jugador(nombre):
    """
    Recibe un nombre (string) y devuelve un diccionario
    con todos los datos iniciales del jugador.
    """
    print(f"[DEBUG] Creando jugador con nombre: {nombre}")

    jugador = {
        "nombre":    nombre,
        "hp":        HP_INICIAL,
        "hp_max":    HP_INICIAL,
        "atk":       ATK_INICIAL,
        "oro":       0,
        "inventario": ["Poción"]
    }

    print(f"[DEBUG] Jugador creado → {jugador}")
    return jugador


# ═══════════════════════════════════════════
# PRUEBA TEMPORAL (lo borraremos después)
# ═══════════════════════════════════════════

print("\n[INFO] Resultado final:")
print(mi_jugador)
👨‍🏫
Ahora noten algo: el [DEBUG] dentro de la función les dice exactamente qué datos entran (nombre) y qué sale (el diccionario completo). Eso les permite verificar que la función hace lo que ustedes creen que hace. Sin ese print, estarían adivinando.

Depuración Aprende a romper las cosas a propósito ▼
👨‍🏫
Hagan esto ahora. Cambien return jugador por simplemente no poner return. Ejecuten. ¿Qué imprime? ¿Por qué? Luego vuélvanlo a poner. Romper el código a propósito es la forma más rápida de entender qué hace cada línea.
Cuando Python no encuentra un return, devuelve None. Si luego intentas acceder a mi_jugador["hp"], Python lanza un TypeError: 'NoneType' object is not subscriptable. Ese error significa exactamente eso: intentaste usar None como si fuera un diccionario.
🐛 Si en algún momento ven NoneType object is not subscriptable en un proyecto real, lo primero que buscan es una función a la que le falta el return.

BLOQUE 02

Mostrar el estado del jugador

separación de responsabilidades · interfaz vs lógica
✓ completado
Objetivo Separar lo que el juego hace de lo que el jugador ve ▼
👨‍🏫
Presten atención a esto porque es un principio que van a usar toda la carrera: una función debe hacer una sola cosa. La función de atacar calcula daño — no imprime nada. La función de mostrar estado imprime — no calcula nada. Si mezclan las dos, cuando quieran cambiar cómo se ve la pantalla, van a tener que tocar la función de combate. Y eso es un desastre.
En ingeniería de software esto se llama principio de responsabilidad única. No es un capricho: cuando los proyectos crecen a miles de líneas, las funciones que hacen muchas cosas se vuelven imposibles de mantener.
Desarrollo Escribe mostrar_estado_jugador ▼
👨‍🏫
Agreguen esta función debajo de crear_jugador. Lean el código antes de copiarlo. Quiero que noten que esta función no devuelve nada. No tiene return. ¿Por qué? Porque su único trabajo es imprimir. No produce ningún dato nuevo.
python
def mostrar_estado_jugador(jugador):
    """
    Recibe el diccionario del jugador y muestra su
    estado actual por pantalla. No devuelve nada.
    """
    print("\n" + "=" * 40)
    print(f"  Héroe:   {jugador['nombre']}")
    print(f"  Vida:    {jugador['hp']}/{jugador['hp_max']}")
    print(f"  Ataque:  {jugador['atk']}")
    print(f"  Oro:     {jugador['oro']}")
    print(f"  Items:   {jugador['inventario']}")
    print("=" * 40)

def crear_enemigo():
    """
    Elige aleatoriamente un enemigo de la lista
    y devuelve su diccionario de datos.
    """
    ENEMIGOS_DISPONIBLES = [
        {"nombre": "Rata Gigante", "hp": 8,  "atk": 2, "oro": 3},
        {"nombre": "Goblin",       "hp": 10, "atk": 3, "oro": 5},
        {"nombre": "Esqueleto",    "hp": 12, "atk": 4, "oro": 7},
    ]

    enemigo = random.choice(ENEMIGOS_DISPONIBLES)
    print(f"[DEBUG] Enemigo generado: {enemigo['nombre']} (HP: {enemigo['hp']})")
    return enemigo


def mostrar_estado_combate(jugador, enemigo):

    mostrar_estado_jugador(jugador)

BLOQUE 04

Las mecánicas de combate

funciones que devuelven valores · mutación de datos · esta_vivo
✓ completado
Objetivo ¿Por qué atacar devuelve el daño en vez de imprimirlo? ▼
👨‍🏫
Imaginen que jugador_ataca() imprimiera el daño directamente. ¿Qué pasa si mañana quieren mostrar el mensaje diferente según si el golpe fue crítico? Tendrían que modificar la función de ataque. Pero si la función devuelve el daño, la función de flujo decide cómo mostrarlo. Las funciones de lógica calculan. Las funciones de interfaz muestran. Nunca las dos.
Desarrollo Cuatro funciones de mecánica ▼
👨‍🏫
Lean cada función antes de copiarla. En especial esta_vivo: recibe cualquier personaje con hp — jugador o enemigo. Eso es posible porque ambos usan la misma clave. Es un ejemplo pequeño de un principio muy poderoso.
python
def jugador_ataca(jugador, enemigo):
    """El jugador golpea al enemigo. Devuelve el daño causado."""
    dano = jugador["atk"] + random.randint(0, 3)
    enemigo["hp"] -= dano
    print(f"[DEBUG] Jugador ataca → daño: {dano} | HP enemigo: {enemigo['hp']}")
    return dano


def enemigo_ataca(jugador, enemigo):
    """El enemigo golpea al jugador. Devuelve el daño causado."""
    dano = enemigo["atk"] + random.randint(0, 2)
    jugador["hp"] -= dano
    print(f"[DEBUG] Enemigo ataca → daño: {dano} | HP jugador: {jugador['hp']}")
    return dano


def esta_vivo(personaje):
    """
    Recibe un jugador O un enemigo (ambos tienen 'hp').
    Devuelve True si sigue vivo, False si no.
    """
    return personaje["hp"] > 0


def dar_recompensa(jugador, enemigo):
    """
    Otorga oro y posiblemente una poción al ganar.
    Devuelve True si hubo poción, False si no.
    """
    jugador["oro"] += enemigo["oro"]
    encontro_pocion = random.random() < 0.5   # 50% de probabilidad

    if encontro_pocion:
        jugador["inventario"].append("Poción")

    print(f"[DEBUG] Recompensa: +{enemigo['oro']} oro | ¿Poción?: {encontro_pocion}")
    return encontro_pocion

La poción y el inventario

condiciones · casos borde · retorno booleano
✓ completado
Objetivo Pensar en todos los casos, no solo el feliz ▼
👨‍🏫
Un programador novato escribe el caso cuando todo va bien. Un programador serio escribe primero los casos cuando algo falla. ¿Qué pasa si el jugador usa la poción y no tiene ninguna? ¿Qué pasa si usa la poción con la vida al máximo? ¿Qué pasa si usa dos pociones seguidas? Esas son las preguntas que deben hacerse antes de escribir la función.
Desarrollo Escribe usar_pocion con todos los casos cubiertos ▼
👨‍🏫
Esta función devuelve True si pudo usarla, False si no. Eso le permite al código que la llama decidir qué hacer en cada caso. Noten también la línea if jugador["hp"] > jugador["hp_max"] — eso es cubrir el caso donde la poción haría superar el máximo.
python
HP_POCION = 8   # ← constante arriba del todo con las demás

def usar_pocion(jugador):
    """
    Si el jugador tiene una Poción, la usa y recupera HP_POCION de vida.
    No puede superar hp_max. Devuelve True si pudo, False si no tenía.
    """
    if "Poción" in jugador["inventario"]:
        jugador["inventario"].remove("Poción")

        hp_antes = jugador["hp"]
        jugador["hp"] += HP_POCION

        # Caso borde: no puede superar el máximo
        if jugador["hp"] > jugador["hp_max"]:
            jugador["hp"] = jugador["hp_max"]

        print(f"[DEBUG] Poción usada → HP: {hp_antes} → {jugador['hp']}")
        return True

    print("[DEBUG] No había poción en el inventario.")
    return False

Los usuarios siempre van a escribir cosas inesperadas — espacios, mayúsculas, letras en vez de números. La función .strip() elimina los espacios al inicio y al final. DANO_HUIDA = 2   # ← va arriba con las constantes

def combate(jugador, enemigo):
    """
    Maneja un combate completo turno a turno.
    Devuelve True si el jugador sobrevivió, False si fue derrotado.
    """
    print(f"\n[INFO] Comienza el combate vs {enemigo['nombre']}!")

    while esta_vivo(jugador) and esta_vivo(enemigo):

        mostrar_estado_combate(jugador, enemigo)
        opcion = mostrar_menu_combate()   # o mostrar_menu_combate(jugador) si lo modificaron

        # ── ATACAR ────────────────────────────────────────────
        if opcion == "1":
            dano = jugador_ataca(jugador, enemigo)
            print(f"  Golpeas y haces {dano} de daño.")

            if not esta_vivo(enemigo):
                print(f"  Derrotaste a {enemigo['nombre']}!")
                encontro = dar_recompensa(jugador, enemigo)
                print(f"  Ganas {enemigo['oro']} de oro.")
                if encontro:
                    print("  Encontraste una poción.")
                break   # ← combate terminado, sale del while

            dano_r = enemigo_ataca(jugador, enemigo)
            print(f"  {enemigo['nombre']} te hace {dano_r} de daño.")

        # ── USAR POCIÓN ───────────────────────────────────────
        elif opcion == "2":
            if usar_pocion(jugador):
                print("  Usas una poción y recuperas vida.")
            else:
                print("  No tienes pociones. Turno perdido.")

            if esta_vivo(enemigo):
                dano_r = enemigo_ataca(jugador, enemigo)
                print(f"  {enemigo['nombre']} te hace {dano_r} de daño.")

        # ── HUIR ──────────────────────────────────────────────
        elif opcion == "3":
            jugador["hp"] -= DANO_HUIDA
            print(f"  Huyes. Pierdes {DANO_HUIDA} HP. HP: {jugador['hp']}")
            break

        # ── OPCIÓN INVÁLIDA ───────────────────────────────────
        else:
            print(f"  Opción '{opcion}' inválida. Elige 1, 2 o 3.")
            # No hay break ni enemigo_ataca → el jugador no pierde el turno

    return esta_vivo(jugador)

Objetivo ¿Qué es if __name__ == "__main__" y por qué importa? ▼
👨‍🏫
Cuando Python ejecuta un archivo directamente, la variable especial __name__ vale "__main__". Cuando otro archivo importa ese módulo, __name__ vale el nombre del archivo. Así que if __name__ == "__main__" es una forma de decir: "esto solo se ejecuta si corren este archivo directamente, no si alguien lo importa". En el bloque siguiente van a importar funciones entre archivos — sin esta línea, todo el juego se ejecutaría solo con importar.

python
def jugar():
    """
    Función principal. Flujo completo: presentación
    → exploración de salas → resultado final.
    """
    # ── PRESENTACIÓN ──────────────────────────────────────
    print("\n" + "=" * 45)
    print("      ⚔  DUNGEON RPG  ⚔")
    print("=" * 45)

    nombre = input("Ingresa el nombre de tu héroe: ").strip()
    if nombre == "":
        nombre = "Héroe Anónimo"
        print("[INFO] Nombre vacío → usando 'Héroe Anónimo'")

    jugador    = crear_jugador(nombre)
    sala_actual = 1

    # ── EXPLORACIÓN DE SALAS ──────────────────────────────
    while sala_actual <= SALAS_TOTALES and esta_vivo(jugador):
        print(f"\n{'─' * 45}")
        print(f"  SALA {sala_actual} de {SALAS_TOTALES}")
        print(f"{'─' * 45}")
        print(f"[DEBUG] Inicio sala {sala_actual} | HP: {jugador['hp']}")

        enemigo    = crear_enemigo()
        sobrevivio = combate(jugador, enemigo)

        if not sobrevivio:
            print(f"[INFO] Jugador derrotado en sala {sala_actual}.")
            break

        print(f"[DEBUG] Sala {sala_actual} completada | HP: {jugador['hp']}")
        sala_actual += 1

        if sala_actual <= SALAS_TOTALES:
            input("\n  [Presiona ENTER para continuar...]")

    # ── RESULTADO FINAL ───────────────────────────────────
    print("\n" + "=" * 45)
    if esta_vivo(jugador):
        print("  VICTORIA — Limpiaste la mazmorra.")
        print(f"  Héroe:  {jugador['nombre']}")
        print(f"  Oro:    {jugador['oro']}")
        print(f"  Vida:   {jugador['hp']}/{jugador['hp_max']}")
        print(f"  Items:  {jugador['inventario']}")
    else:
        print("  DERROTA — Caíste en las sombras.")
        print(f"  Llegaste hasta la sala {sala_actual} de {SALAS_TOTALES}.")
    print("=" * 45)


# ─── PUNTO DE ENTRADA ──────────────────────────────────────
if __name__ == "__main__":

Refactorización en módulos

carpetas · import · separación de responsabilidades
✓ completado
Objetivo ¿Por qué no queda todo en un archivo? ▼
👨‍🏫
Su archivo tiene ahora unas 150 líneas. Parece manejable. Pero un proyecto real puede tener 10.000 líneas. Si todo está en un archivo, ¿cómo encuentran la función que quieren modificar? ¿Cómo saben qué parte puede romper otra? La solución es dividir por responsabilidades: un archivo para crear cosas, uno para las mecánicas, uno para la interfaz, uno para el flujo. Cada archivo hace una sola cosa. Si mañana quieren cambiar cómo se ven los menús, solo abren interfaz.py.
La regla de oro de la refactorización: el juego debe funcionar exactamente igual antes y después. Si cambia el comportamiento, cometiste un error. Solo cambia la organización.
Desarrollo La nueva estructura de carpetas ▼
👨‍🏫
Creen una carpeta nueva llamada dungeon_rpg/. Dentro, esta estructura. El archivo __init__.py puede estar vacío — su sola existencia le dice a Python que esa carpeta es un módulo.
text

├── main.py               ← solo llama a jugar()
└── game/
    ├── __init__.py       ← vacío
    ├── creacion.py       ← crear_jugador, crear_enemigo
    ├── interfaz.py       ← mostrar_estado, mostrar_menu
    ├── mecanicas.py      ← atacar, usar_pocion, esta_vivo, recompensa
    └── flujo.py          ← combate(), jugar()
